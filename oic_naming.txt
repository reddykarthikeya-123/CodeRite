OIC Naming Conventions and Best PracticesNaming conventionsGeneral GuidelinesAvoid abbreviations: Abbreviations may be ambiguous. The names used must be spelled out. Do not abbreviate unless the object name becomes too long. Avoid abbreviations unless they are commonly accepted system abbreviations (ERP, HCM, AR, AP, SF etc.).Artifact names must be alphanumeric and underscore only. Names must be composed only of alphanumeric characters. Words comprising a name should be concatenated with underscore (_) or Pascal Case fashion. Example: PurchaseOrderAvoid using numeric characters in the name unless it is required to convey some business meaning. No special characters such as spaces, '.', '$', '%', '#', []Indicate artifact type in the name to reduce ambiguity. When the same name is used for different artifact types, append a suffix to indicate its type. It makes it easier to distinguish these artifacts by identifying their types: <Artifact Name><Type Suffix>Example: for country look up, COUNTRIES_LOOKUPProject Naming convention<CUST_PREFIX>_<MODULE_NAME>_PROJECTCUST_PREFIX: Name of client in short form (for ex: Oceaneering, Hanwha)MODULE_NAME: Name of the module like HCM, AR, AP etc.Common project<CUST_PREFIX>_COMMON_PROJECTThis common project serves as a centralized foundation to host shared connections, reusable utilities, and standard components across all OIC integrations running on other projects.Integration Flow namesFormat:<CUST_PREFIX>_<INT_NUM>_<SOURCE>_<TARGET>_<DATA_OBJECT>_< OPERATION>_<TYPE>CUST_PREFIXUnique short code for the Customer name (for ex: Hanwha)It is recommended to  follow the same convention across the OIC artifactsA unique company name can be used as a prefix to prevent naming collisions with other libraries or projectsINT_NUM: [Optional]Unique integration identifier assigned in CEMLI to easily identify the integrationThis is an optional, can be added if the customer wants to identify the OIC integration flows by unique number Ex: 001 – Customers, 002-Items, 003-Supplier etc.Sub flows: Use the parent flow Integration number in the dedicated child flowsCommon Flows: Use a generic code or COMMON/UTIL keywordSOURCE: Source system Name. Example: Scheduler - (SCH or SCHEDULER)PollerMainFlowAPISUBPublisherListenerCOMMONGENERICConnectionFormat: <CUST_PREFIX>_<SYSTEM>_<ADAPTER_NAME>CUST_PREFIX: Name of client in short form (for ex: Hanwha)SYSTEM: Name of the system to which we are connecting using the connection. This leads to many duplicate connections.Antipattern: Every developer creates their own connection using a different set of configurations/credentials.Why an Antipattern? : High number of connections make manageability painful, especially when you need to update the endpoint, credentials, configuration, and so on.Complicates impact analysis when there is an application upgrade or metadata/coordinate change.Best Practice: Have a custodian create needed connections and ensure duplicate connections of the same types are not created.Build a best practice for naming conventions and maintaining a set of configurations.Scheduled Integration that Never Stops Trying to ProcessProcess records within a set of files with a tight service level agreement (SLA). For example, synchronize employee records into Active Directory from Oracle HCM Cloud or send benefits information).Antipattern: The scheduled integration looks for all files to process and loops over all, to sequentially process until no files remain.Why an Antipattern? : If a large number of files exist, one run of a scheduled integration executes for a long time and starves other integrations and may get terminated by the framework.Processing tied to a single server, which does not leverage multiple nodes in a cluster.Best Practice: Limit the number of files to process in a single scheduled run. To do this, Limit results from the list file.Use schedule parameters to remember the last processed file for the next run.Invoke the run now command from the same integration to trigger processing of the next file if waiting for the next scheduled run is not feasible.Synchronous Integration Doing Too MuchA request triggers complex processing involving enrichment and updates across multiple systems.Antipattern: Huge synchronous integrations modeling a large number of invokes /conditional logic.Synchronous integrations with invokes within a loop with a large number of iterations.Why an Antipattern? : Susceptible to timeouts - Any marginal slowdown adds up.Blocking call – Holds resources and starves other integrations.Best Practice: Explore moving completely to an asynchronous integration i.e. fire and forget or asynchronous response. In this case, OIC platform provides an acknowledgment to clients upon receiving the message. OIC ensures guaranteed processing and supports resubmission of failures.Split into a synchronous integration containing mandatory processing before sending out a response and triggering separate asynchronous fire and forget integrations for other processing logic.Optimize synchronous processing with a coarse-grained external API to replace multiple chatty calls.Read Files with Many RecordsRead a file with a large number of records and process individual records.Antipattern: Reading the entire file in memory using the read file option and processing record by record.Why an Antipattern? : Consumes large amounts of memory and impact other system processing.Sequential processing does not leverage built-in map reduction capabilities.Best Practice: Download the file to the stage location using the download file option. Stage files are secured and deleted at the completion of the instance.Use the read file with segments option. You can still model record level processing inside the segment processing.OIC automatically processes segments in parallel.OIC brings in only portions of the file to memory, as needed.Chatty IntegrationsSynchronize records in a file or large data set with an external system (for example, synchronizing journal transactions or uploading employee records into Oracle HCM Cloud).Antipattern: Use an invoke activity within a looping construct to call external APIs for every record.Why an Antipattern? : Downstream applications are receiving a large number of atomic requests. This puts the entire system under duress.A usage-based pricing model translates to high costs.Best Practice: Leverage application capabilities to accept multiple records in a single request. For example, Salesforce: 200 records, Oracle Engagement Cloud/ERP Cloud: 100 records, Service Cloud: 1000 recordsLeverage adapter capabilities to send a large data set as attachments/files. For example, Salesforce Adapter: 10,000 records/10 MB files, Oracle ERP Cloud Adapter supports FBDI filesUse a stage file action (append to file option) to send the file to the destination at the end.Import an Externally Updated IAR FileNeed to leverage advanced XSL constructs that may not be available in the mapper.Antipattern: Updating the IAR file externally and then importing it into Oracle Integration.Why an Antipattern? : This can lead to metadata inconsistency and validation failures. And also activation failures may occur.Best Practice: Use the import map feature in Oracle Integration. This will keep metadata consistent and leverages validation.Integrations Running Unchanged Despite Changing Business NeedsIntegrations/schedules created during the initial implementation continue to run even though your business requirements have changed over time.Antipattern: Integrations and scheduled jobs created during the initial product implementation are never re-evaluated against changing business needs.A proliferation of integrations occurs without looking at existing integrations.Why an Antipattern? : Unnecessary runs of jobs that handle no work.Unoptimized invocations of multiple integrations with similar functionality.Clutter with dead integrations, life cycle management (LCM) overheads, and developer confusion.Best Practice: Periodically analyze existing integrations/schedules against current business needs. Look at monitoring data for historical runs.Consolidate integrations that are very similar. Check all flows to find out if there are more than one integration exist to perform the same activity.Deactivate integrations that are no longer relevant. An integration for generating a file meant for partner consumption becomes irrelevant when the partner no longer needs it.Adjust the schedules for integrations that need lesser frequency and delete schedules no longer required.Files generated monthly have schedules run every few minutes.Changes in business needs can make certain schedules unnecessary.Scheduled Orchestration IntegrationsAlways decouple Scheduling logic with Business logic using an Async Handoff from Scheduled Integration Flows. This will ensure that the scheduler threads are not used for completing the entire Business logic.Scheduled Orchestrations are meant to serve particular requirements of scheduling flows, and freeing them up using Async-handoff makes the solution scalable and performant when processing large number of requests.Scheduled orchestrations should not be used as a substitute for App Driven orchestrations.Synchronous IntegrationsA synchronous integration calling any asynchronous request response service:Calling asynchronous fire and forget (one-way) is acceptable.Oracle Integration does not currently allow modeling an asynchronous request response service. All scheduled orchestration patterns internally use an asynchronous request response.A synchronous integration calling multiple services that span more than 5 minutes is reported as a stuck thread in Oracle WebLogic Server.Estimate file sizesMake sure to estimate the response size/file size in OIC integration flow to avoid running into file size exceeds errors in OIC.
